# 会悟本地会议记录系统

会议记录是会悟从“语音转写工具”走向“会议知识工作台”的持久化边界。它保存会议级信息、分段音频、说话人、识别元数据和摘要状态，使实时录音、文件上传、记录检索和大模型整理使用同一份可追溯数据。

## 设计思路

早期实现只保存会议和文本分段，无法覆盖实时麦克风、分段音频和停止录音后的最终说话人对齐。当前分支保留了“会议级 + 分段级”的关系模型，但把它接入会悟现有的完整工作流：

```text
创建草稿
  → 麦克风或上传产生分段
  → 增量保存文本、音频与识别元数据
  → 停止录音后写入 canonical final transcript
  → finalize 汇总全文、时长、说话人和统计
  → 记录管理检索 / 编辑 / 导出 / 生成摘要
```

这样可以避免长会议结束时一次性提交全部音频，也避免最终识别结果与已保存记录脱节。

## 本地数据

默认数据库为 `backend/data/records.db`，可通过历史兼容变量 `DITING_RECORDS_DB_PATH` 覆盖。

| 数据 | 内容 |
|---|---|
| `meeting_records` | 标题、来源、原文件、说话人、完整文本、时长、状态和所有者 |
| `meeting_record_segments` | 顺序、时间、说话人、文本、识别元数据和 WAV 音频 BLOB |
| 摘要记录 | 生成状态、Markdown、结构化内容和脱敏模型配置快照 |
| 摘要条目 | 按类型保存议题、决策、行动项、风险或结构图节点 |

列表和搜索只读取会议级字段，不加载大体积音频。打开详情、播放分段或下载源音频时才读取 BLOB。

## API 生命周期

### 记录

- `POST /api/records`：创建记录草稿。
- `GET /api/records?q=关键词`：列出或搜索当前用户记录。
- `GET /api/records/{record_id}`：读取详情及分段。
- `GET /api/records/{record_id}?include_audio=true`：读取详情时同时恢复全部分段 WAV；记录管理的“载入转写界面”使用此模式。
- `PUT /api/records/{record_id}`：更新标题、文本和整理结果。
- `DELETE /api/records/{record_id}`：删除单条记录。
- `GET /api/records/{record_id}/text`：导出纯文本。
- `GET /api/records/{record_id}/source-audio`：读取或下载源音频。
- `POST /api/records/{record_id}/finalize`：汇总分段并完成记录。

### 分段

- 新录音或上传任务可逐段写入记录，避免在浏览器长期持有全部音频。
- 分段更新保持顺序、时间轴、说话人和音频引用一致。
- 最终转写完成后，以 canonical transcript 更新草稿，而不是与实时 partial 重复追加。

### 摘要

- `POST /api/record-summaries`：为记录创建摘要任务。
- `GET /api/record-summaries` 与 `GET /api/record-summaries/{summary_id}`：查询摘要列表和详情。
- `/api/record-summaries/{summary_id}/retry`、`/download`：重试或下载摘要。
- 摘要失败可重试，历史结果仍可查看或下载。
- 摘要始终读取已定稿记录的 `full_text`；模型字段先按 JSON Schema 校验并归一化，课程、讲座、访谈或演示不会因为缺少项目行动项而被误判为空内容。
- 没有实际内容的章节不会渲染，剩余章节连续编号。
- 文字流程图由结构化节点生成 Mermaid/Markdown，不依赖图像模型。

接口沿用会悟账号鉴权。普通用户只能访问自己的记录；管理员按权限查看全局数据。删除、更新和摘要任务都会再次校验所有者。

## 界面流程

1. 登录会悟，在“会议转写”点击“新建记录”。
2. 使用麦克风或上传音频；需要多人日志时启用说话人选项。
3. 录音期间查看 X-ASR 实时预览，停止后等待所选最终引擎定稿。
4. 修改标题、说话人或分段文字，点击“保存记录”。
5. 在“记录管理”检索、载入、继续整理、导出、删除或生成摘要。

快捷键 `Ctrl+H` 可打开或关闭记录管理面板。

## 数据一致性原则

- 完整 WAV 是会后重算和故障恢复的事实来源。
- `final_transcript` 是最终文字事实来源，实时 partial 只用于会中反馈。
- ASR、说话人、分段音频与波形使用同一 16 kHz 时间轴。
- 摘要引用已定稿记录，不直接依赖仍在变化的实时缓冲。
- 列表不加载音频；用户载入某条历史记录时完整读取该记录的分段音频，恢复真实波形与播放接口。WaveSurfer 不可用时显示占位波形，并使用浏览器原生音频兜底播放。

## 兼容与迁移

`DITING_RECORDS_DB_PATH` 仅为历史兼容变量；正式产品与界面名称统一为“会悟”。已有 `records.db` 可以直接继续使用，不应为了品牌改名删除或重建数据库。未来若调整表结构，应使用向前迁移并保留旧记录可读性。
